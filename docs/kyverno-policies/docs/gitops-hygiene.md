# GitOps Hygiene Policies

Category: `gitops-hygiene` · 4 policies · Wave 1 · PRD-2026-KYVERNO-POLICY-001

These policies keep the GitOps contract honest: everything running in a tenant namespace should be traceable to a Git commit, an owning tenant, and a deterministic sync order. They are defense-in-depth alongside ArgoCD's own RBAC and AppProject controls.

---

## require-ownership-labels

**Severity: medium · Wave 1 · Test: `tests/gitops-hygiene/require-ownership-labels/`**

Requires `platform.example.com/tenant` and `platform.example.com/cost-center` labels on all workload kinds (Deployment, StatefulSet, DaemonSet, CronJob). These labels drive the Datadog tenant namespace dashboards, quota chargeback, and violation-report attribution — an unlabeled workload is invisible to all three.

**How to comply:** The Copier onboarding templates stamp these labels into the Kustomize base; workloads created through standard onboarding already comply. Violations indicate hand-rolled manifests outside the template.

**Placeholder warning:** Label keys must be aligned with the real platform label schema (the one the quota dashboards already query) before merge.

## require-argocd-tracking

**Severity: high · Wave 1 · Test: `tests/gitops-hygiene/require-argocd-tracking/`**

Requires the ArgoCD tracking label (`app.kubernetes.io/instance`) on tenant-labeled workloads. A tenant workload without it was applied outside GitOps — `kubectl apply` straight to the cluster — which means it's invisible to drift detection, won't survive a sync-prune, and bypasses PR review entirely.

In Audit mode this is the platform's **kubectl-bypass detector**; in Enforce it closes the bypass path outright.

**How to comply:** Deploy through your ArgoCD Application. There is no compliant way to hand-apply tenant workloads, by design. Break-glass operational changes should go through the documented emergency process, not silent kubectl.

**Verify before Enforce:** Confirm the Akuity instance's tracking method (label vs. annotation tracking) — if the instance uses annotation tracking, this policy must check `argocd.argoproj.io/tracking-id` instead.

## require-sync-wave-on-hooks

**Severity: low · Wave 1 · Test: `tests/gitops-hygiene/require-sync-wave-on-hooks/`**

Requires an explicit `argocd.argoproj.io/sync-wave` annotation on ArgoCD-managed Jobs in tenant namespaces (CronJob-spawned Jobs are excluded via ownerReferences). Jobs are almost always ordering-sensitive — migrations, schema setup, cache warms — and relying on default wave 0 produces race conditions that only surface intermittently. This codifies the sync-wave conventions already established in the CFK and PingIDM deployment patterns.

**How to comply:**
```yaml
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "-1"   # run before the workloads that depend on it
```

## restrict-appproject-scope

**Severity: high · Wave 1 · Test: `tests/gitops-hygiene/restrict-appproject-scope/`**

Denies tenant AppProjects with wildcard `sourceRepos` or wildcard destination namespaces. A wildcard AppProject lets any Application under it deploy from any repo to any namespace — collapsing the tenant isolation boundary that AppProjects exist to hold. Platform-owned projects (`default`, `platform`, `platform-system`) are excluded by name.

**How to comply:** Tenant AppProjects are generated by the onboarding automation with explicitly scoped repos and destinations; this policy should only fire on hand-edited projects.

**Why this exists alongside ArgoCD RBAC:** ArgoCD RBAC controls *who* can modify AppProjects; this controls *what* an AppProject is allowed to say regardless of who wrote it. Defense in depth — either control failing alone is survivable.

**Test prerequisite:** The Chainsaw test requires the ArgoCD AppProject CRD in the test cluster (apply it during kind cluster bootstrap).
