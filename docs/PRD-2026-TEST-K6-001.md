## 1. Overview

This document defines requirements for a platform load and regression testing capability built on Grafana k6, integrated into the AKS-based GitOps platform. The system provides two execution modes: lightweight smoke and regression gates executed directly in Azure DevOps pipelines, and distributed in-cluster load runs executed via the k6 Operator and reconciled through ArgoCD. All test results are exported to Datadog via OpenTelemetry for correlation with existing gateway and tenant observability.

The primary near-term driver is the Envoy Gateway migration (PRD-2026-INGRESS-001): the system must produce a defensible latency and error-rate baseline of the current ingress path before cutover, and an equivalent post-cutover measurement, so that migration validation is data-driven rather than anecdotal.

> **Scope boundary:** k6 validates traffic paths — request latency, error rates, and throughput SLOs. It does not validate Kubernetes state, policy behavior, or GitOps reconciliation. Those concerns belong to admission policy testing (PRD-2026-KYVERNO-POLICY-001) and platform observability workstreams.

## 2. Problem Statement

The platform supports approximately 100 tenants across dev, nonprod, and prod environments. Changes to the request path — Envoy Gateway migration, `ClientTrafficPolicy` changes, Cilium network policy rollout (PRD-2026-NETPOL-ZT-001), Kyverno admission webhook configuration — currently ship without automated performance regression detection. Three specific gaps:

1. **No ingress baseline.** The Envoy Gateway migration will change the external traffic path (Imperva CWAF → Envoy). Without a measured baseline of the current Kong/App Gateway path, there is no objective way to confirm the new path performs equivalently or better, and post-cutover tenant complaints cannot be triaged against pre-cutover data.
2. **No regression gate for request-path changes.** Configuration changes to gateway policies, network policies, or admission webhooks that sit in the data path or add per-request latency can degrade tenant experience silently. Detection today is reactive (tenant reports, Datadog anomalies).
3. **No synthetic validation of tenant provisioning.** The Copier-based onboarding system provisions tenant routes, but there is no automated end-to-end traffic check confirming a newly onboarded tenant's ingress path actually serves traffic within SLO.

## 3. Goals and Non-Goals

### Goals

- **G-1:** Establish a Git-versioned k6 test suite with pass/fail thresholds declared in code, executable both in ADO pipelines and in-cluster.
- **G-2:** Produce a reproducible ingress-path baseline for the Envoy Gateway migration and a like-for-like post-cutover comparison methodology.
- **G-3:** Gate request-path configuration changes (gateway, network policy, admission webhook) behind an automated smoke/regression stage in ADO.
- **G-4:** Export all k6 metrics to Datadog with consistent tagging (`test_type`, `environment`, `route`, `test_run_id`) for dashboarding and run-over-run comparison.
- **G-5:** Contain load-test blast radius via a dedicated namespace with resource quota, scoped RBAC, and network policy.

### Non-Goals

- **NG-1:** Load testing tenant application logic. Tenants own performance testing of their own workloads; this system may later offer them a scaffold (Phase 4) but does not test on their behalf.
- **NG-2:** Chaos or fault-injection testing.
- **NG-3:** Production load testing. All load-profile runs target dev and nonprod. Production receives at most low-rate synthetic smoke checks, subject to OQ-3.
- **NG-4:** Soak/endurance testing beyond the steady-state windows defined in regression scenarios (may be revisited post-Phase 3).
- **NG-5:** Replacing Datadog synthetics for uptime monitoring. k6 runs are change-triggered or scheduled test events, not continuous monitors.

## 5. Success Metrics

- **SM-1:** Ingress baseline captured for the current path (external and internal variants) before Envoy cutover, with at least three runs demonstrating < 10% run-over-run variance in steady-state p95.
- **SM-2:** 100% of ingress-path configuration changes (defined path filter in ADO) pass through the k6 smoke gate before nonprod promotion by end of Phase 2.
- **SM-3:** Post-cutover Envoy comparison completed and documented as part of PRD-2026-INGRESS-001 validation, with p95/p99 delta and error-rate delta quantified per route.
- **SM-4:** Smoke gate P50 pipeline overhead ≤ 3 minutes; false-positive (flake) rate < 5% over a rolling 30-day window.
- **SM-5:** Zero tenant-impacting incidents attributable to load-test execution (quota breach, noisy-neighbor effects) — measured continuously.

## 6. Architecture

### 6.1 Components

| Component | Role |
|---|---|
| `k6/` directory in platform repo | Git-versioned scenarios (`scenarios/`), shared config (`lib/`), operator manifests (`operator/`) |
| ADO pipeline stages | `k6_smoke` (container job, `grafana/k6` image, no cluster dependency) and `k6_regression` (triggers in-cluster TestRun and waits) |
| k6 Operator | In-cluster distributed execution; `TestRun` CRD with `parallelism` for multi-pod runner distribution |
| `platform-loadtest` namespace | Execution boundary: ResourceQuota, scoped Role/RoleBinding, Cilium network policy permitting egress only to gateway targets and the Datadog Agent |
| Kustomize `configMapGenerator` | Ships scenario scripts to the cluster; hash suffix forces TestRun re-evaluation on script change. Multi-file suites bundled via `k6 archive` |
| Datadog Agent OTLP endpoint | Receives k6 OpenTelemetry output (gRPC :4317); metrics tagged for dashboard filtering and run comparison |
| ArgoCD Application | Reconciles operator install and `k6/operator/` manifests; TestRun triggering remains pipeline-owned (see 6.3) |

### 6.2 Execution modes

**Mode A — CI smoke gate (ADO container job).** Small deterministic load (constant-vus, ~30s) against dev. Threshold breach yields nonzero exit code, failing the stage with no additional gating logic. Runs on every change matching the ingress-path filter.

**Mode B — In-cluster regression run (k6 Operator).** Ramping load profile with a steady-state comparison window, executed as a `TestRun` with distributed runners. Triggered by the ADO pipeline post-deploy to nonprod; the pipeline applies the TestRun, waits on `status.stage=finished`, evaluates runner exit codes, and deletes the TestRun.

### 6.3 GitOps boundary

The operator installation and namespace scaffolding are ArgoCD-managed. TestRun lifecycle is deliberately pipeline-owned rather than Git-committed: TestRuns are ephemeral jobs, and keeping trigger authority in the pipeline colocates it with the pass/fail gate and the CR evidence trail. This mirrors the reconciler-purity principle of PRD-2026-CI-RENDER-001 — ArgoCD reconciles desired state; imperative test execution is a pipeline concern.

### 6.4 Traffic identification

All synthetic traffic carries `X-Synthetic-Test: k6-platform` and `X-Test-Run-Id: <build id>` headers so that Imperva rules, NSG flow log analysis, and tenant dashboards can distinguish synthetic from real traffic. Scenarios support `TARGET_PATH=external` (through Imperva CWAF) and `TARGET_PATH=internal` (cluster-local Envoy service) to isolate the boundary hop.

## 7. Functional Requirements

### FR-1: Test suite structure and versioning

- **FR-1.1:** All k6 scenarios, shared libraries, and threshold definitions SHALL reside in the platform repository under a `k6/` directory and be subject to standard PR review.
- **FR-1.2:** Every scenario SHALL declare pass/fail thresholds in code (`options.thresholds`); threshold breach SHALL produce a nonzero exit code.
- **FR-1.3:** Scenarios SHALL read target environment (`TARGET_ENV`), path variant (`TARGET_PATH`), and run identifier (`TEST_RUN_ID`) from environment variables; no environment-specific values SHALL be hardcoded in scenario logic.
- **FR-1.4:** All scenarios SHALL apply the standard tag set: `test_type`, `environment`, and per-request `route` tags.

### FR-2: CI smoke gate

- **FR-2.1:** An ADO stage SHALL execute the smoke scenario in a `grafana/k6` container job (pinned version) against dev with no cluster credentials required.
- **FR-2.2:** The stage SHALL be wired to a path filter covering ingress-path configuration (gateway resources, `ClientTrafficPolicy`, relevant Cilium and Kyverno policy paths).
- **FR-2.3:** The stage SHALL publish the k6 summary export as a build artifact.
- **FR-2.4:** The smoke scenario SHALL verify both availability (status checks) and path identity (response served by the expected gateway).

### FR-3: In-cluster regression execution

- **FR-3.1:** The k6 Operator SHALL be installed as a system workload, reconciled by ArgoCD.
- **FR-3.2:** Regression runs SHALL execute as `TestRun` resources with configurable `parallelism`, applied by the ADO pipeline post-deploy to nonprod.
- **FR-3.3:** The pipeline SHALL wait on TestRun completion with an explicit timeout, derive stage pass/fail from runner pod exit codes, and delete the TestRun on completion regardless of outcome.
- **FR-3.4:** Scenario scripts SHALL be delivered via Kustomize `configMapGenerator` with hash suffixing; multi-file suites SHALL be bundled with `k6 archive`.

### FR-4: Ingress baseline and comparison (Envoy migration)

- **FR-4.1:** A regression scenario SHALL exercise a representative set of tenant routes through the current ingress path, in both external and internal variants, producing a tagged baseline (`test_run_id=baseline-<change>`).
- **FR-4.2:** The identical scenario SHALL be replayable against the Envoy path post-cutover (`test_run_id=post-<change>`) with no scenario-code differences other than environment inputs.
- **FR-4.3:** Baseline validity SHALL require a minimum of three runs with steady-state p95 variance under 10% (SM-1) before it is accepted as the comparison reference.
- **FR-4.4:** A documented comparison procedure SHALL define the steady-state window, the metrics compared (p95, p99, error rate, per route), and the acceptance criteria feeding the PRD-2026-INGRESS-001 go/no-go decision.

### FR-5: Observability integration

- **FR-5.1:** k6 runners SHALL export metrics via OpenTelemetry (OTLP gRPC) to the Datadog Agent; the deprecated StatsD output SHALL NOT be used.
- **FR-5.2:** A Datadog dashboard (standalone or a group appended to PRD-2026-DD-TENANT-NS-001 scope) SHALL present p95/p99 by route, error rate, and VU count, filterable by `test_run_id` for A/B comparison.
- **FR-5.3:** The dashboard SHALL overlay Envoy Gateway upstream latency against k6-observed latency to isolate the Imperva/NSG boundary hop.

### FR-6: Blast-radius containment

- **FR-6.1:** All in-cluster test execution SHALL run in a dedicated `platform-loadtest` namespace with a ResourceQuota sized to prevent tenant workload starvation.
- **FR-6.2:** Pipeline RBAC SHALL be scoped to the `platform-loadtest` namespace via Role/RoleBinding; cluster-scoped credentials SHALL NOT be used for test triggering.
- **FR-6.3:** A Cilium network policy SHALL restrict runner egress to gateway targets and the Datadog Agent endpoint.
- **FR-6.4:** The namespace SHALL be excluded from Kyverno HPA-injection policies and any other mutation policies that would conflict with operator-managed pod lifecycles.

### FR-7: Tenant onboarding smoke (Phase 3)

- **FR-7.1:** The tenant onboarding flow SHALL support an optional post-provisioning synthetic check that exercises the new tenant's route and validates response within SLO.
- **FR-7.2:** Onboarding smoke failures SHALL surface in the provisioning ledger with the failing check detail.

## 8. Non-Functional Requirements

- **NFR-1:** Smoke gate wall-clock overhead ≤ 3 minutes at P50 (SM-4).
- **NFR-2:** Regression stage total timeout explicitly declared (initial: 30 minutes) per platform pipeline timeout conventions.
- **NFR-3:** k6 image and operator versions pinned and upgraded via PR; no `latest` tags.
- **NFR-4:** Test scripts SHALL NOT contain secrets; any authenticated-route testing uses ESO-sourced credentials mounted to runner pods (Phase 3+, subject to OQ-4).
- **NFR-5:** Load profiles for nonprod SHALL be documented with expected peak RPS and reviewed against shared-infrastructure capacity (Imperva, NSG flow limits) before first execution.

## 9. Phases

### Phase 1 — CI smoke gate and baseline scaffolding

**Scope:** `k6/` directory structure, shared config library, smoke scenario, ADO `k6_smoke` stage against dev, Datadog OTLP output validated from CI runs, tagging conventions established.

**Exit gate:** Smoke gate green on three consecutive ingress-path changes; k6 metrics visible in Datadog with correct tags; SM-4 flake criteria measurable.

### Phase 2 — In-cluster execution and Envoy migration baseline

**Scope:** k6 Operator install via ArgoCD, `platform-loadtest` namespace with quota/RBAC/network policy, regression scenario, `k6_regression` ADO stage, ingress baseline runs captured and validated per FR-4.3, comparison procedure documented.

**Prerequisite:** Phase 1 exit gate met.

**Exit gate:** Accepted baseline on record for the current ingress path (external and internal); regression stage demonstrated end-to-end (trigger → wait → evaluate → cleanup) in nonprod; blast-radius controls verified (quota enforcement tested deliberately).

### Phase 3 — Migration validation and onboarding smoke

**Scope:** Post-cutover Envoy comparison runs and documented delta analysis feeding PRD-2026-INGRESS-001; tenant onboarding synthetic check integrated with the Copier flow and provisioning ledger.

**Prerequisite:** Phase 2 exit gate met; Envoy cutover scheduled under PRD-2026-INGRESS-001.

**Exit gate:** SM-3 satisfied; onboarding smoke live for new tenant provisions in nonprod.

### Phase 4 — Tenant scaffold (optional / deferred)

**Scope:** Copier template component offering tenants an opt-in `k6/` scaffold with platform tagging conventions and namespace-quota guidance baked in.

**Prerequisite:** Phase 3 exit gate met; demand signal from at least two tenant teams.

## 10. Constraints and Assumptions

- **C-1:** Four-engineer team with ~10 concurrent PRD workstreams; this PRD is intentionally phased so Phase 1 delivers value in roughly a day of effort and later phases can pause without stranding work. (Directly responsive to the maturity assessment finding on design velocity vs. enforcement velocity.)
- **C-2:** Datadog Agent OTLP gRPC receiver availability is assumed; if not yet enabled, enabling it is a Phase 1 task, not a blocker.
- **C-3:** Load-profile runs are restricted to dev/nonprod (NG-3). Nonprod shares boundary infrastructure (Imperva, NSGs) with other consumers; NFR-5 review is mandatory before first ramp run.
- **C-4:** The k6 Operator manages runner pods directly; assumption that excluding the namespace from mutation policies (FR-6.4) is sufficient to avoid reconciliation conflicts — to be verified in Phase 2.
- **C-5:** Baseline capture (Phase 2) must complete before Envoy cutover; the migration schedule in PRD-2026-INGRESS-001 is therefore a sequencing constraint on this PRD, not the reverse.

## 12. Open Questions

- **OQ-1:** Which tenant routes constitute the "representative set" for the ingress baseline (FR-4.1)? Candidates: highest-traffic tenants per Datadog, plus one route per distinct gateway policy configuration. Needs selection criteria and sign-off.
- **OQ-2:** Should the smoke gate be blocking or advisory during Phase 1 calibration? Precedent from the AI PR review PRD suggests advisory-first with a documented flip criterion — proposed: advisory for two sprints or until flake rate < 5%, whichever is later.
- **OQ-3:** Is any production synthetic smoke in scope (low-rate, post-change verification), or is production coverage delegated entirely to Datadog synthetics? Interacts with change-management evidence requirements.
- **OQ-4:** Do any baseline-relevant routes require authentication? If so, ESO-sourced credential mounting moves from Phase 3+ into Phase 2 scope.
- **OQ-5:** Does Imperva rate-limiting or bot mitigation require an explicit allowlist entry for the synthetic-traffic header/source before external-path ramp runs? Coordination owner needed.
- **OQ-6:** Should regression results attach to ServiceNow CRs via the GitOps-to-ITSM change-bridge as supporting evidence? If yes, define the artifact format and bridge extension in a follow-on revision.
