# 04 — Supply Chain Trust: Signing, Provenance, and Identity

**Parent:** ARCH-2026-CI-ARTIFACT-001 Principle P3 (hermetic builds, mandatory provenance); cross-cutting across Build, Registry, Promotion, and Admission planes
**Feature prefix:** `SCT`

---

## Executive Summary

Trust in this architecture is cryptographic and identity-based, never positional: no artifact is believed because of where it sits or which pipeline produced it, only because of what can be verified about it. Every Release Bundle carries a Sigstore keyless signature bound to the producing pipeline's OIDC identity, plus a SLSA provenance attestation generated by platform-controlled machinery isolated from user build code. Verification is enforced at three independent boundaries — promotion intake, and cluster admission being the two hard gates, with registry-side checks as an early tripwire. There are **no long-lived signing keys anywhere in the system**; identity is short-lived certificates from a certificate authority tied to the CI's OIDC token, with every signing event recorded in a tamper-evident transparency log. This document defines the trust model, the private-vs-public Sigstore infrastructure decision, the verification policy language, and the rollout from audit to enforce.

**Why it matters to the business:** this is the control set that turns supply-chain compliance (SLSA levels, provenance mandates, audit trails) from an annual evidence-gathering scramble into a queryable property of the system — and it is the backstop that makes every other component's compromise survivable.

## Principle Breakdown

### P3 — Provenance is mandatory; verification is everywhere
1. **Keyless signing only.** Signing uses ephemeral keys certified against the pipeline's OIDC identity (issuer + subject). Nothing to rotate, leak, or exfiltrate; identity revocation = identity provider action. Long-lived keys are prohibited by policy and by the absence of any component that accepts them.
2. **Provenance from the platform, not the build steps.** Provenance generation runs in machinery the tenant's build code cannot influence (isolated/reusable-workflow-equivalent in the CI system) — the property that distinguishes SLSA L2/L3 from self-attested L1.
3. **Verify at every boundary; each verifier is independent.** Promotion intake (Doc 06) and admission (Doc 07) each verify signature identity, provenance subject/builder/source match, and attestation freshness against their own trust roots. A single compromised verifier does not collapse the chain.
4. **Transparency log as the audit spine.** Every signature is recorded; monitors alert on signatures from unexpected identities over org artifacts (a canary for CI compromise).

### Trust model (what verification actually checks)

For a bundle at digest D, a verifier requires:
- **Signature:** valid Sigstore signature over D; certificate identity matches allow-listed pattern (`<ci-issuer>` + subject matching `repo:<org>/<repo>:ref:refs/heads/main` class patterns); certificate chained to the org's trusted root; entry present in transparency log within certificate validity.
- **Provenance:** in-toto SLSA v1 predicate whose subject digest = D's image/config digests; builder ID = the platform's isolated provenance generator; source URI + commit match bundle annotations; build type in the approved set.
- **Freshness:** vulnerability-scan attestation timestamp within policy window (e.g., 7 days for prod promotion; re-scan attaches a new referrer without changing D).
- **Policy report:** policy-evaluation attestation present and produced against a policy-bundle version ≥ minimum currently mandated.

### Infrastructure decision: private Sigstore stack
Recommend a **private deployment** of the certificate authority (Fulcio-equivalent) and transparency log (Rekor-equivalent), federated to the org's identity provider:
- Public-good Sigstore publishes signing metadata to public logs — internal repo names, workflow paths, and cadence become public. For an enterprise, private infrastructure with identical protocols preserves the trust properties without the exposure.
- The trust root (CA cert, log public key) is distributed to verifiers via the platform repo and rotated on a scheduled ceremony.
- Timestamp authority included so signature validity outlives certificate lifetime.

## Risks
- **Private Sigstore operational burden** — real but bounded; mitigate with managed-service evaluation as a fallback in the ADR, and by treating the stack as a Tier-0 platform service with the same SLOs as the registry.
- **Verification latency at admission** — mitigate with verifier-local caching keyed by digest (immutability makes cache invalidation trivial) and pre-verification at promotion so admission is a re-check, not first contact.
- **Freshness gates blocking emergency deploys** — break-glass path defined in GOV-05 with mandatory post-hoc review; break-glass usage is itself attested and alarmed.
