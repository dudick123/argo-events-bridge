# 03 — Build Plane

**Parent:** ARCH-2026-CI-ARTIFACT-001 §4.2, Principle P4 (CI produces; CD promotes; the cluster verifies)
**Feature prefix:** `BLD`

---

## Executive Summary

The Build Plane is a single logical pipeline per application repository with exactly four phases — build, render, validate, publish — and exactly one side effect: pushing a signed Release Bundle to the registry. CI holds no cluster credentials, has no environment awareness beyond render targets, and cannot trigger a deployment. This document details each phase, the hermeticity and determinism requirements, the validation gate battery, and the versioned pipeline-template mechanism that lets ~100 tenant repos share one implementation. Deliverables span Phase 0–1: template v1 producing conformant bundles for two pilot runtimes, expanding to the full runtime matrix by Phase 2, with hermetic-build hardening continuing into Phase 4.

**Why it matters to the business:** one shared, versioned pipeline for the entire fleet means a security fix or capability lands everywhere with one release — and stripping deployment power from CI removes the largest, most commonly exploited credential surface in modern delivery systems.

---

## Principle Breakdown

### P4 — Strict separation of produce / promote / verify
- The pipeline identity can *only* push to specific registry paths and sign via OIDC. It cannot reach cluster APIs, promotion APIs, or environment configuration. Blast radius of a fully compromised pipeline: publishing a malicious artifact — which then fails provenance/policy verification at promotion (Doc 06) and admission (Doc 07). Defense in depth is structural.
- No "deploy" stage exists to remove. There is nothing to disable in an incident; the pipeline simply cannot act on environments.

### Phase-by-phase design

**1. Build (hermetic, attested)**
- All dependencies resolved from pinned lockfiles via an internal proxy/cache; no ambient internet during compile. Network egress from build steps is deny-by-default (SLSA L3 posture direction).
- Two supported build strategies behind one interface: Cloud Native Buildpacks (default; platform-managed base images and patching) and Dockerfile (escape hatch; policy-linted). Provenance requirements identical for both — generated by platform-controlled machinery isolated from user build steps, never by user code (the composite-vs-isolated distinction that separates trustworthy provenance from theater).
- Base images: a curated, internally rebuilt set with its own bundle lifecycle; tenant builds pin by digest and receive automated uplift PRs.

**2. Render (deterministic, versioned toolchain)**
- Maps typed config source (Doc 02) → plain manifests using a **render toolchain that is itself a pinned, versioned artifact** (container image with fixed tool versions). Toolchain upgrades are releases with fleet impact reports (SRC-06), never silent behavior changes.
- Output passes the determinism canonicalizer (RB-03). Same inputs ⇒ same config-layer digest; this property is tested in CI, not assumed.
- Per environment-class rendering emits the small family of config artifacts under one release version.

**3. Validate (rendered output, pre-merge and pre-publish)**
- Schema/API conformance including CRDs via generated JSON schemas.
- **Admission-policy pre-check:** the exact policy bundle the clusters enforce (Doc 09) executed against rendered output in CI — admission failures are found at PR time, never at sync time. Policy bundle version recorded in the bundle's policy-report attestation.
- Secret leakage scan over rendered output and source.
- Semantic diff vs. currently promoted digest per environment-class, posted to the PR (feeds Doc 08).
- Gate results are emitted as a signed policy-evaluation attestation attached to the bundle (RB-04) — validation isn't just a gate, it's evidence.

**4. Publish (the only side effect)**
- Assemble bundle (RB-02), sign keyless with the pipeline's OIDC identity (Doc 04), attach attestations, push, verify-after-push (pull back and validate structure + signatures before reporting success).

## Risks
- **Runtime matrix breadth delays GA** — mitigate by strict pilot-first sequencing; the template interface, not runtime coverage, is the Phase 1 exit criterion.
- **Hermeticity friction (private/native deps)** — proxy allow-list process with SLA; hermeticity level tracked per-repo and ratcheted, not big-banged.
