# Skill 8 — Datadog Query & Dashboard Builder

**Purpose:** Translate plain-language observability questions into correct Datadog queries, monitors, and dashboard widgets that follow platform tagging and dashboard conventions — so output is consistent with the tenant dashboard standard instead of one-off snowflakes.

## Step-by-step

### 1. Scope the skill
- In scope: metric/log/trace query construction, widget JSON authoring, monitor definition drafting, tag-taxonomy guidance, "which metric answers this question" mapping.
- Out of scope: Datadog account administration, agent installation debugging, cost/ingestion optimization (candidate future skill).
- Primary users: platform engineers and tenants building views over their namespaces.

### 2. Gather ground truth
- Your tag taxonomy: the canonical tags (tenant, environment, cluster, namespace, service, team), their exact key names, and which are reliable vs. sparsely applied. Include tag cardinality warnings.
- Your metric inventory by source: what KSM/custom-resource-state emits, OpenMetrics-scraped metrics, agent-native metrics — with the actual metric names as they appear in your org (prefixes and pipelines rename things; generic names will be wrong).
- The tenant dashboard PRD's conventions: layout standards, template variables, naming, what a conforming dashboard contains.
- Your monitor conventions: naming, tagging, notification routing, evaluation windows.
- A set of validated golden queries: 10–15 real queries the team trusts, each labeled with the question it answers.

### 3. Draft the skeleton
- SKILL.md: question-to-signal mapping guide (which telemetry answers which question class) → query construction rules with the tag taxonomy → dashboard conventions → monitor conventions → output formats (query string, widget JSON, monitor JSON).
- `references/`: full tag taxonomy, metric inventory by source, golden query library, dashboard PRD conventions summary.
- Description triggers: "datadog", "dashboard", "monitor for", "alert when", "query for pod restarts", "show me metrics", tenant/namespace observability phrasing.

### 4. Encode the golden-query pattern
Instruct the skill to compose new queries by adapting the nearest golden query rather than constructing from scratch — adaptation from validated examples produces far fewer subtly-wrong queries (wrong aggregation, wrong tag key) than free synthesis. When no golden query is close, the skill says so and marks the output as needing validation.

### 5. Declare blind spots
- The skill cannot execute queries or verify a metric exists in your org. Instruct it to mark every metric name it wasn't given in the inventory as "verify in metric explorer first", and never to invent plausible-sounding metric names — a query against a nonexistent metric silently returns nothing, which is worse than an error.
- Tag taxonomy drift: state the inventory's as-of date.

### 6. Add worked examples
Three end-to-end: a plain-language question → signal selection reasoning → final query → widget JSON. Choose examples covering the three main sources (KSM, OpenMetrics, agent-native) since query syntax differs.

### 7. Test triggering and outputs
- Negatives: Prometheus/Grafana questions (legacy — decide whether to redirect or answer), log *pipeline* configuration, Azure Monitor questions.
- Output test: verify generated queries against the live org (the one skill where output testing means actually running the result) — check metric names resolve and tags exist.

### 8. Maintain via inventory regeneration
The metric inventory and tag taxonomy are the drift surfaces. Script their extraction from the Datadog API so the reference files regenerate on a schedule instead of decaying by hand.

## Definition of done
- [ ] Metric inventory uses actual org metric names, regenerated by script
- [ ] Golden query library seeded with validated queries
- [ ] Skill adapts from golden queries; flags free-synthesized output for validation
- [ ] Hard rule against inventing metric names
- [ ] Output conforms to tenant dashboard PRD conventions
